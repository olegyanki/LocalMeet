# ADR-003: Оптимізований RPC замість N+1 запитів

## Статус: Прийнято

## Контекст

Початкова реалізація списку чатів робила N+1 запитів:
- 1 запит на список чатів
- N запитів на учасників кожного чату
- N запитів на останнє повідомлення
- N запитів на unread count

Для 10 чатів = 31 запит. Це повільно і створює навантаження.

## Рішення

Один RPC `get_my_chats_optimized` який повертає все одним запитом:
- Чати з деталями
- Учасники (як масиви)
- Останнє повідомлення
- Unread count
- Інформація про подію (walk_title, walk_image_url, etc.)

Аналогічно `get_chat_details` для одного чату.

## Альтернативи

1. **Promise.all для паралельних запитів** — швидше ніж послідовні, але все одно N+1
2. **Денормалізація** (зберігати last_message в chats) — складніше підтримувати
3. **GraphQL** — overkill для нашого масштабу

## Наслідки

- 1 запит замість 31 для 10 чатів
- Складніший SQL, але простіший клієнтський код
- Потрібно оновлювати RPC при зміні схеми
- Типи генеруються автоматично з Supabase

- `supabase/migrations/*_group_chat_rpc_functions.sql`
- `src/shared/lib/api/chats.ts` → `getMyChats()`
